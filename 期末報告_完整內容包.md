# 第二部分：報告核心內容（可直接改寫成正文）

## 第 1 章　緒論

釣魚（Phishing）是目前佔比最高、成本最低、卻最難防的攻擊手法之一——它攻擊的不是程式漏洞，而是**人的判斷**。對計算機概論的學生而言，這個主題特別適合：釣魚網站的「製作」其實只用到我們這學期學的 HTML、CSS、JavaScript 與表單；它的「偵測」則可以用 Python 的字串處理與規則邏輯實現。也就是說，**同一套基礎工具，可以拿來理解攻擊，也可以拿來建立防禦**——這正是資安的核心張力。

本報告以「釣魚網站」與「釣魚插件（瀏覽器擴充功能）」為主軸，輔以社交工程的心理機制，並全程採教育/防禦立場：所有示範皆為本機模擬，目的在於讓讀者**看懂攻擊長相、進而學會辨識與防禦**。

## 第 2 章　社交工程與釣魚的基本原理

### 2.1 技術面：釣魚的攻擊鏈

一次釣魚攻擊通常包含四個階段：

1. **誘餌投遞（Lure）**：透過郵件、簡訊（Smishing）、即時通訊或廣告，把受害者導向惡意連結。
2. **偽裝呈現（Spoof）**：一個與真實服務視覺幾乎相同的登入頁。
3. **憑證擷取（Capture）**：受害者輸入帳密，被表單送往攻擊者。
4. **變現與橫向移動（Exploit）**：用竊得憑證登入真實服務、盜用、或當作入侵組織的跳板。

### 2.2 心理面：為什麼聰明人也會上當

社交工程利用的是人類認知的「捷思」（heuristics）。常見的六個心理觸發器（取自 Cialdini 影響力原則，套用到資安情境）：

| 心理觸發器 | 釣魚中的具體話術 |
|---|---|
| **權威（Authority）** | 「這是銀行/系統管理員/IT 部門通知…」 |
| **急迫（Urgency / Scarcity）** | 「您的帳號將於 24 小時內停用」 |
| **恐懼（Fear）** | 「偵測到異常登入，請立即驗證」 |
| **信任 / 熟悉（Familiarity）** | 冒用你認識的品牌、同事或主管名義 |
| **貪婪（Greed）** | 「您有一筆退款 / 中獎，請點此領取」 |
| **從眾 / 一致（Commitment）** | 「您先前已同意更新資料，請完成驗證」 |

> 教學要點：**急迫感是最強的單一訊號**。任何要你「立刻、馬上、否則就…」的訊息，都應該先停下來、改用官方管道自行查證。

## 第 3 章　釣魚網站的技術實現方式（防禦視角拆解）

> 本章說明攻擊「如何運作」，目的是讓你**辨識**它，而非製作它。隨附的 `demo_a` 即為此章的本機教學模擬。

### 3.1 URL 偽造：最關鍵的破綻

釣魚網站再像，**網址不會說謊**。常見偽造手法與對應的判讀方法：

| 偽造手法 | 範例 | 如何識破 |
|---|---|---|
| 子網域冒名 | `paypal.secure-login.xyz` | 從右往左讀，真正網域是 `secure-login.xyz` |
| 拼字仿冒（Typosquatting） | `g00gle.com`、`app1e.com` | 用數字替字母 |
| 同形異義（IDN Homograph） | 用西里爾字母的「а」替拉丁「a」 | 瀏覽器顯示 Punycode（`xn--…`） |
| `@` 混淆 | `https://google.com@evil.xyz` | 瀏覽器只認 `@` 之後的主機 |
| 裸 IP | `http://192.168.x.x/login` | 正規服務不會用 IP 當登入頁 |
| 高風險 TLD | `.xyz / .tk / .top / .zip` | 免費/低價網域是釣魚溫床 |

### 3.2 視覺複製：HTML/CSS 為何能被「秒複製」

任何網頁的前端原始碼都是公開的——攻擊者只要「另存網頁」或檢視原始碼，就能取得完整的 HTML/CSS，在幾分鐘內重建一個視覺上一模一樣的頁面。**這帶出本報告最重要的防禦觀念**：

> 頁面長得像不像，與它安不安全，**完全是兩回事**。判斷真偽唯一可靠的依據是「網址 + 來源驗證機制」，而不是「看起來像」。

### 3.3 表單與 JavaScript 行為

正常登入頁的 `<form>` 會把資料送往該服務自己的後端；釣魚頁則把 `action` 指向攻擊者的伺服器，或用 JavaScript 監聽輸入框即時側錄。`demo_a` 用本機鏡射演示了「攻擊者能看到你打的每個字」這件事，但**刻意不實作任何外傳程式碼**。

## 第 4 章　釣魚插件的攻擊與防禦概念

瀏覽器擴充功能（Browser Extension）能讀取、修改你瀏覽的每一個頁面，是高權限、低警覺的攻擊面。

### 4.1 惡意擴充功能的威脅模型（原理，非製作教學）

- **過度索取權限**：安裝時要求 `"<all_urls>"`、`tabs`、`webRequest` 等廣泛權限。
- **內容腳本注入**：透過 content script 在你打開的銀行/信箱頁面注入程式，讀取表單或竄改畫面。
- **供應鏈風險**：合法擴充被**轉售或被駭**後推送惡意更新——使用者毫無察覺。
- **常見惡意行為**：側錄輸入、竊取 Cookie/Session、置換廣告、把流量導向釣魚頁。

### 4.2 防禦型擴充功能與使用者自保

- **防禦型擴充**：如即時 URL 信譽檢查、反釣魚黑名單比對、密碼管理員（依網址自動填入，網址不符就拒填——這是極強的反釣魚機制）。
- **使用者最佳實踐**：
  1. 只從官方商店安裝，安裝前看**權限清單**與評價。
  2. 對要求「讀取所有網站資料」的擴充提高警覺。
  3. 定期清查、移除不再使用的擴充。
  4. 企業端可用**擴充白名單**政策（Chrome Enterprise / Edge Policy）。

> 教學要點：密碼管理員之所以能反釣魚，正是因為它**綁定網域**——假網站網址不符，它就不會自動填密碼，等於替你做了一次 URL 比對。

## 第 5 章　生成式 AI 對釣魚的雙面影響

（本章可大量沿用你原本《生成式 AI 對資訊安全的雙面影響》報告的論述，並把案例從「郵件」換成「網站/插件」。）

### 5.1 攻擊面：AI 如何降低釣魚門檻

- **內容生成**：自動產生無語病、語氣道地的多語言誘餌文案，破解「看錯字抓釣魚」的舊經驗。
- **頁面複製加速**：把目標頁面截圖或原始碼丟給 AI，可快速生成視覺相似的前端骨架。
- **規模化與客製化**：結合外洩資料做「魚叉式釣魚」（Spear Phishing），千人千面。

> ⚠️ 立場聲明：上述是**已被資安研究普遍記錄的現象**，本報告僅說明「威脅變化」，不示範如何優化攻擊。

### 5.2 防禦面：AI 如何增強偵測

- **語義偵測**：規則只看結構（URL、關鍵字），AI 能讀懂「情境與話術」——例如判斷一段文字是否在製造急迫與冒用權威。
- **可疑頁面分析**：把 URL 與特徵交給 AI 做綜合研判（見 `demo_b` 與 `phishing_analyzer.py` 的 AI 協作區）。
- **資安教育**：把艱深技術翻譯成一般使用者能懂的語言，提升整體警覺。

### 5.3 軍備競賽下的結論

AI 同時加速了攻與防。**決定勝負的不再是技術門檻，而是使用者的判斷力與組織的防禦縱深**——這也是為什麼「教育」始終是最關鍵的一道防線。

## 第 6 章　實際防禦技術與最佳實踐

採「三層縱深」框架呈現：

**個人層**
- 從右往左讀網域；對「急迫/恐嚇」訊息一律改用官方管道查證。
- 使用密碼管理員（綁定網域、自動拒填假站）。
- 開啟**多因素驗證（MFA）**，優先用 FIDO2／實體金鑰（可抗釣魚）。

**組織層**
- 員工反釣魚演練與教育訓練（提升通報率）。
- 郵件閘道過濾（SPF / DKIM / DMARC）、擴充功能白名單政策。
- 最小權限原則、定期憑證輪替。

**技術層**
- 規則式偵測（如本報告 Demo）＋ AI 語義偵測雙軌。
- 瀏覽器內建的 Safe Browsing / SmartScreen 黑名單。
- HTTPS 與憑證透明度（CT log）監控品牌被冒名的網域註冊。

## 第 7 章　結論與對計算機概論學生的啟示

1. **同一套基礎技術可攻可守**——HTML/JS 能做釣魚頁，也能做偵測工具；差別在於意圖與倫理。
2. **網址是唯一可靠的真相**——再逼真的頁面也騙不過正確的 URL 判讀與綁定網域的驗證機制。
3. **防禦的最後一哩是人**——技術會被 AI 拉平，判斷力才是護城河。
4. **倫理是工程師的底線**——學會攻擊原理是為了防禦，跨越法律界線只需一行外傳程式碼，但代價是刑責。

# 第三部分：網頁 Demo 操作說明

## Demo A：釣魚網站模擬器 + 即時警示系統（`demo_a_釣魚網站模擬器.html`）

**直接用瀏覽器打開即可，無需安裝。** 畫面分為「兩個現實」：

- **左側（誘餌）**：虛構品牌 *CloudVault* 的登入頁，刻意做得「看起來可信」。
- **右側（攻擊者視角）**：當你在左側打字，右側的攻擊者主控台會**即時鏡射**你輸入的帳密與鍵盤側錄。
- **頂部**：仿瀏覽器網址列，標出「真正的網域不是 CloudVault」這個破綻。
- **下方**：四張「破綻卡」教你該注意什麼，以及「檢視原始碼」按鈕——展示攻擊者如何輕易複製頁面。

**安全設計**：全程本機執行，無 `fetch`、無 `form action`、零外傳；品牌純屬虛構。

**現場演示腳本（約 2 分鐘）**：
1. 先讓觀眾看左側登入頁，問：「這頁看起來可信嗎？」
2. 在帳號欄打字 → 指著右側：「攻擊者此刻已看到你打的每個字。」
3. 按「登入」→ 出現「攻擊已完成（模擬）」說明轉址話術。
4. 開「檢視原始碼」→ 強調：「複製一個頁面只要幾分鐘，所以不能靠『看起來像』判斷安全。」

## Demo B：釣魚 URL 偵測工具（`demo_b_釣魚URL偵測工具.html`）

輸入網址 → 即時給出 0–100 風險分數與**逐項判斷理由**（命中哪些特徵、各加幾分）。內建可點擊的範例網址。底部「請 AI 幫我分析」會**自動產生一段分析 Prompt** 供你複製貼到 AI 助手（刻意不內嵌 API 金鑰）。

**現場演示腳本（約 1.5 分鐘）**：依序點 `google.com`（綠燈 0 分）→ `paypal-secure.account-update.xyz`（紅燈 70 分），對比說明「分數差距來自哪些特徵」。

# 第七部分：現場操作具體建議

1. **先離線測試**：兩個 HTML 直接雙擊用瀏覽器開即可，不需網路；建議用 Chrome/Edge 全螢幕示範。
2. **Demo A 用「對比」說故事**：左右螢幕同框最有衝擊力——「你以為在登入，攻擊者同步在收割」。
3. **Demo B 用「分數跳動」製造張力**：先綠燈再紅燈，讓觀眾看到分數與理由的變化。
4. **AI 協作要「秀過程」**：別只貼結果，現場把 Prompt 貼進 Claude/ChatGPT，讓老師看到你會「引導 AI」。
5. **備援方案**：把每個 Demo 的關鍵畫面預先截圖放進簡報，萬一現場環境出狀況可直接看圖講解。
6. **時間分配**：原理 5 分鐘、Demo 6 分鐘、AI 協作 2 分鐘、結論 2 分鐘。
